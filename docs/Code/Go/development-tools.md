# Go 开发工具

本篇聚焦 Go 开发环境的选择和基础配置：VS Code、GoLand、gopls、Delve、`go fmt` 与 `go vet`。
后续“Go 语言开发工具”页面可深入性能分析、覆盖率、代码生成、构建标签和 CI，本篇避免完全重复。

## 工具分层

一个可靠的 Go 环境可分为四层：

- Go SDK：编译、测试、格式化和基础分析的来源。
- 语言服务器 gopls：补全、跳转、重构和编辑器诊断。
- 编辑器或 IDE：组织界面和调用底层工具。
- 调试器 Delve：断点、单步、变量和调用栈检查。

编辑器不是编译器。IDE 显示正常但命令行构建失败时，应以模块目录中的 `go` 命令结果为准。

## VS Code 配置

安装 Microsoft 的 VS Code 后，从扩展市场安装官方 Go 扩展 `golang.go`。
打开模块根目录，而不是只打开单个 `.go` 文件；根目录通常包含 `go.mod`。

可在工作区设置中启用保存时格式化：

```json
{
  "[go]": {
    "editor.defaultFormatter": "golang.go",
    "editor.formatOnSave": true
  },
  "gopls": {
    "ui.semanticTokens": true
  }
}
```

不要复制大量不理解的 gopls 实验选项。先使用扩展默认值，只对团队确认的问题增加工作区设置。

## GoLand 配置

GoLand 是集成式商业 IDE，通常能自动识别 SDK 和 `go.mod`。
在设置中确认 Go SDK 版本、项目模块根目录、格式化和保存动作。
团队不应依赖只存在于某位开发者 GoLand 配置中的构建参数；可复现命令应写入脚本、Makefile 或 CI。

## gopls

gopls 是 Go 官方语言服务器，可由编辑器自动安装，也可手工安装：

```bash
go install golang.org/x/tools/gopls@latest
gopls version
```

安装后的命令通常位于 `go env GOPATH` 下的 `bin` 目录。
编辑器找不到 gopls 时，先检查该目录是否在 PATH，再检查编辑器使用的 Go SDK。

gopls 常见能力包括：

- 基于类型信息的补全和诊断。
- 定义、引用、实现和调用层次跳转。
- 导入整理、重命名和部分代码操作。
- 对多模块工作区的 `go.work` 支持。

## Delve 调试器

Delve 的命令是 `dlv`：

```bash
go install github.com/go-delve/delve/cmd/dlv@latest
dlv version
dlv debug .
```

在调试会话中可设置断点、单步、查看局部变量和调用栈。
优化和内联可能影响调试体验；开发调试通常让 Delve 管理合适的编译参数，不要把禁用优化的构建当作发布构建。

## `go fmt` 与 gofmt

格式化当前模块所有包：

```bash
go fmt ./...
```

格式化指定文件可直接使用：

```bash
gofmt -w main.go
```

`go fmt` 按包调用格式化工具，`gofmt` 更适合文件级操作和检查差异。
Go 项目不应手工争论空格、制表符和大括号风格，提交代码应保持 gofmt 结果。

## `go vet`

```bash
go vet ./...
```

`go vet` 检查可疑结构，例如部分格式化参数错误、不可复制锁值和错误标签。
它不是完整静态分析器，也不证明程序正确；通过 vet 不代表没有数据竞争、资源泄漏或业务漏洞。

## 差异示例一：VS Code 与 GoLand

VS Code 较轻量、扩展化，适合希望组合工具和统一跨语言编辑器的团队。
GoLand 集成更深，重构和数据库等功能集中，但需要授权并消耗更多资源。
两者都应调用同一个模块工具链，代码质量不能依赖特定 IDE 才成立。

## 差异示例二：格式化与静态诊断

`go fmt ./...` 改写代码布局，不负责判断业务逻辑。
`go vet ./...` 报告部分可疑代码，通常不负责改变源码。
CI 中通常同时运行两者，并额外运行测试；不能用其中一个替代另一个。

## 最小验证流程

```bash
go version
go env GOMOD
gopls version
dlv version
go fmt ./...
go vet ./...
go test ./...
```

若仓库要求格式化检查而不允许 CI 自动改文件，可比较 `gofmt -d` 输出或使用项目既有脚本。

## 常见错误

- 同时安装多个 Go 扩展，导致格式化器和语言服务器冲突。
- 编辑器终端与图形进程继承了不同 PATH，看到不同 Go 版本。
- 只打开子目录，导致 gopls 找不到模块或工作区上下文。
- 把所有 gopls 诊断都关闭来隐藏真实编译问题。
- 在生产二进制上远程开放无保护调试端口。
- 认为 `go vet` 可以替代测试、竞态检测或安全审查。

## 工程实践

- 工作区只保存团队需要的设置，个人主题和字体不提交到项目配置。
- 编辑器保存时格式化，CI 再验证格式和测试，形成双重保障。
- 调试配置不包含密钥、生产地址或个人绝对路径。
- 工具升级应与 Go 版本升级分开评估，便于定位诊断变化。
- 以命令行命令作为可复现基线，IDE 只负责提高交互效率。

完成配置后，应能在任一编辑器中获得一致的格式化、诊断、跳转和本地调试体验。
