# HTML 转义、XSS 防护与 CTF 安全分析
本篇只讨论无害演示、代码审计和防护。所有实验应限定在自己编写的页面、赛事附件、
本地靶场或明确授权环境中。示例不会提供可直接滥用的 XSS 载荷。

## 安全问题的核心边界

模板需要把“程序生成的结构”和“外部数据”组合成响应。如果数据被浏览器解释为标记、
脚本、样式或危险 URL，它就越过了数据与代码边界。XSS（跨站脚本）是这类边界失守
的一组结果，而不是某一个固定字符串或单一过滤规则。

防护主线是验证业务数据但不依赖输入过滤，默认启用模板自动转义，按**输出上下文**
选择编码或安全 API，避免危险解析入口，并以 CSP、Trusted Types、Cookie 属性纵深防御。

## 字符编码先于转义
服务器应明确发送 UTF-8：

```http
Content-Type: text/html; charset=utf-8
```

HTML 前部也声明：

```html
<meta charset="utf-8">
```

编码把字节解释为 Unicode 字符；转义把某些字符表示为当前语法中的普通数据。
若客户端与服务端对编码理解不一致，过滤器、模板和浏览器可能面对不同字符序列。
统一 UTF-8、拒绝非法编码并避免重复解码，是可靠处理的基础。

## HTML 实体与字符引用

在 HTML 文本中，有语法意义的字符可以写成字符引用：

```html
<p>用户输入示例：&lt;学习笔记&gt; &amp; 草稿</p>
```

浏览器显示为“`<学习笔记> & 草稿`”，但不会把尖括号内容解析成元素。
常见引用如下：

| 字符 | 引用 |
| --- | --- |
| `&` | `&amp;` |
| `<` | `&lt;` |
| `>` | `&gt;` |
| `"` | `&quot;` |
| `'` | `&#39;` |

应对原始数据编码一次再交给浏览器。重复编码会显示 `&lt;` 字样，过早解码则可能
重新暴露语法字符。

## 输出上下文决定规则

“把尖括号替换掉”不是完整方案。相同数据可能进入不同解析器：

1. HTML 文本节点。
2. 带引号的 HTML 属性值。
3. URL 及其查询参数。
4. JavaScript 字符串或程序结构。
5. CSS 值或选择器。

每种上下文的终止符、转义规则和安全 API 都不同。一个上下文中的正确编码不能直接
复制到另一个上下文。最佳策略通常是减少跨上下文拼接，而不是维护复杂手写替换表。

## HTML 文本上下文

服务端模板应默认自动转义：

```html
<p>昵称：{{ display_name }}</p>
```

这里 `{{ display_name }}` 代表某模板系统的占位符，具体语法因框架而异。
只要未关闭自动转义，数据中的 `&`、`<` 等会作为文字显示。

在浏览器端显示纯文本时优先使用：

```js
output.textContent = untrustedText;
```

不要为了插入普通文字而使用 `innerHTML`。

## HTML 属性上下文

属性值始终加引号，并由模板引擎执行属性上下文转义：

```html
<input name="nickname" value="{{ nickname }}">
```

转义只能确保数据不突破当前引号，不能使任意属性本身变安全。例如事件处理属性会把
值交给 JavaScript 解析器，`style` 会进入 CSS 上下文，某些 URL 属性会触发导航。
因此不要把不可信值放入属性名、标签名、事件处理属性或原始样式中。

布尔属性不应写成来自用户的任意字符串，应由程序逻辑决定是否输出该属性。

## URL 上下文

查询参数应使用 URL API 编码，而不是手工拼接：

```js
const url = new URL("/search", location.origin);
url.searchParams.set("q", untrustedText);
link.href = url.href;
```

URL 编码解决的是分隔符和数据表示，不会判断目标是否可信。若用户可以提供完整链接，
应用还应解析 URL，限制协议、主机或路径范围，并拒绝用户名密码、异常端口等非业务形式。
不要通过简单大小写比较或字符串前缀判断 URL 安全性。

## JavaScript 上下文

尽量不要把数据直接嵌入内联脚本：

```html
<script>
  // 不推荐在此处手工拼接模板数据。
</script>
```

更稳妥的方式是通过安全序列化的 JSON 数据块、DOM 属性或网络 API 传递数据，再由外部
脚本读取。JSON 序列化与 HTML 嵌入是两个层次：即使值是合法 JSON，也必须保证它不会
提前结束所在 HTML 元素。应使用框架提供的专用“安全 JSON 嵌入”能力。

JavaScript 字符串转义不能替代 HTML 转义，反过来也不成立。

## CSS 上下文

不要把不可信字符串拼进 `style` 属性、`style` 元素或样式表。颜色、尺寸和主题等值应
映射到预定义类名或严格枚举。CSS 转义只解决语法表示，不保证属性和值安全。

## 安全 DOM API 与危险入口

优先使用把数据当作文本或明确属性的 API：

```js
const item = document.createElement("li");
item.textContent = label;
item.setAttribute("data-id", stableId);
list.append(item);
```

需要重点审计的入口包括 `innerHTML`、`outerHTML`、`insertAdjacentHTML`、
`document.write`，以及接受代码字符串的执行接口。它们不一定永远禁止使用，
但输入必须来自受信任常量或经过成熟、配置正确的 HTML 清理器处理。

不要用正则表达式尝试完整解析和清理 HTML；HTML 的容错和上下文规则远比标签匹配复杂。

## XSS 类型

反射型问题把请求数据带入当前响应，存储型问题先保存再展示，DOM 型问题由客户端脚本
把 URL、消息或存储等来源送入危险入口。分类不改变防护原则；审计应追踪从来源
（source）到危险入口（sink）的完整数据流，而不是只搜索某个关键词。

## CSP 基础

内容安全策略（Content Security Policy）通过 HTTP 响应头限制页面可加载和执行的资源：

```http
Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'none'; frame-ancestors 'none'
```

`default-src` 设置默认来源，`script-src` 控制脚本并可采用 nonce 或 hash，`object-src`
禁用旧式插件，`base-uri` 限制 URL 基准，`frame-ancestors` 控制页面能否被嵌入。

CSP 应先使用 `Content-Security-Policy-Report-Only` 观察兼容性，再逐步收紧。
策略需结合站点实际资源、第三方依赖、nonce/hash 生命周期和报告采集设计。

CSP 是纵深防御，不应替代上下文转义、DOM API 选择和富文本清理。
依赖宽泛来源、`'unsafe-inline'` 或不受控第三方脚本会明显削弱策略。

## 输入验证的正确角色
输入验证用于确保数据符合业务约束：类型、长度、字符范围、枚举、对象归属和状态转换。
例如用户名可限定为合理长度，但不能因为通过正则就直接安全嵌入所有输出上下文。

不要全局删除尖括号来“防 XSS”：它可能破坏合法内容，也无法覆盖属性、URL、脚本和
DOM 入口。富文本应交给维护良好、严格允许列表配置的 HTML 清理器；清理后不要再次
字符串拼接。保存原始还是规范化数据应按业务决定，输出安全应在靠近输出处保证。

## CTF 中的安全分析方法

仅在授权题目中，可以按以下无害流程分析：

1. 阅读题目范围，确认目标、允许工具和数据处理边界。
2. 用查看源码比较服务端初始 HTML，用元素面板观察最终 DOM。
3. 在网络面板记录输入进入哪个请求字段、响应位置和内容类型。
4. 使用普通标记文本，如 `<学习主题>`，观察它被显示、编码还是删除。
5. 查找客户端验证，并确认服务端是否独立执行同等或更严格验证。
6. 审计脚本的数据来源与 DOM 入口，记录完整数据流。
7. 给出修复建议：安全 API、上下文转义、清理器、CSP 和服务端验证。

无需构造可执行载荷也能定位多数缺陷。不得测试范围外系统或收集会话、个人数据、凭据。

## 防护审计清单

- HTTP 与文档字符编码是否一致且为 UTF-8。
- 模板自动转义是否开启，是否存在无理由的“安全 HTML”绕过标记。
- 每个输出点是否按 HTML、属性、URL、JavaScript 或 CSS 上下文处理。
- 客户端是否优先使用 `textContent`、`createElement` 等安全 DOM API。
- 富文本是否由维护良好的允许列表清理器处理。
- 完整 URL 是否经过解析与协议/来源限制，而非简单字符串匹配。
- CSP 是否通过响应头部署，策略是否避免不必要的宽泛许可。
- 会话 Cookie 是否使用适当的 `HttpOnly`、`Secure` 和 `SameSite` 属性。
- 服务端是否独立验证数据和权限，不信任 hidden、disabled 或前端校验。
- 日志、错误页和调试信息是否避免泄露令牌与敏感数据。
- 自动化测试是否覆盖特殊字符、空值、超长值和编码边界。
