# HTML 知识体系

HTML（HyperText Markup Language，超文本标记语言）是描述网页结构与语义的**标记语言**，
不是编程语言。它通过元素、属性和文本告诉浏览器“内容是什么”，CSS 负责主要视觉表现，
JavaScript 负责主要交互行为。三者可以协作，但职责不应混淆。

HTTP 响应体和屏幕上的渲染结果，因为安全分析与浏览器调试都依赖这些边界。

## 最小 HTML5 文档

```html
<!doctype html>
<html lang="zh-CN">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>页面标题</title>
  </head>
  <body>
    <h1>页面主标题</h1>
    <p>这是一个结构完整的 HTML5 文档。</p>
  </body>
</html>
```

- `<!doctype html>` 让浏览器使用标准模式解析页面。
- `<html lang="zh-CN">` 声明文档语言，辅助读屏、翻译和搜索处理。
- `<meta charset="utf-8">` 应尽早出现，声明字节如何解码为文本。
- viewport 声明让移动浏览器按设备宽度布局。
- `<title>` 是标签页标题，也是书签与搜索结果的重要来源。
- `<body>` 包含呈现给用户的主要文档内容。

## 元素、标签与属性

元素通常由开始标签、内容和结束标签组成：

```html
<p class="summary">一段摘要。</p>
```

这里 `<p>` 与 `</p>` 是标签，整体是一个 `p` 元素，`class` 是属性。
某些空元素不能包含子内容，例如 `img`、`input`、`meta`、`link` 和 `br`：

```html
<img src="diagram.png" alt="请求与响应流程图">
```

HTML 语法不像 XML 那样要求空元素以 `/>` 结束；在 HTML 文档中写 `>` 即可。
属性值建议始终加引号，以减少空格、特殊字符和模板拼接造成的歧义。

## 树结构心智模型

元素通过嵌套形成树。标签应按相反顺序闭合：

```html
<article>
  <h2>标题</h2>
  <p>正文包含 <strong>重点</strong>。</p>
</article>
```

浏览器会容错修复部分错误标记，但修复结果不一定符合作者预期。
“页面看起来正常”不能证明结构有效，更不能证明不同浏览器会得到相同 DOM。

## 结构与表现分离

优先让 HTML 表达语义，让 CSS 表达布局和颜色：

```html
<p class="warning">操作前请确认目标环境已获授权。</p>
```

不要为了变红而选择一个语义不相关的元素，也不要依赖大量 `<br>` 制造间距。
类名宜表达角色，如 `warning`、`site-nav`，而不是 `red-text`、`left-20px`。

## 语义带来的价值

恰当的 `nav`、`main`、`article`、标题、列表和表单控件可以同时改善：

- 浏览器和搜索引擎对页面结构的理解。
- 键盘用户与读屏软件用户的导航体验。
- CSS 选择器和自动化测试的稳定性。
- 团队阅读、维护和复用代码的效率。

语义不是“多用 HTML5 标签”这么简单。若元素已有正确含义，就不应先用 `div`，
再用 ARIA 模拟它。原生 `button` 通常优于带点击事件的 `div`。

## HTML、CSS 与 JavaScript 的边界

```html
<button id="toggle" type="button">显示详情</button>
<section id="details" hidden>
  <h2>详情</h2>
  <p>这是可切换的内容。</p>
</section>
```

HTML 定义按钮与内容区，CSS 可以设计样式，JavaScript 可以切换 `hidden` 状态。
即使脚本暂时失败，文档也应尽量保留可理解的结构和基础功能，这称为渐进增强。

## 可访问性最低基线

- 为页面设置准确的 `lang`。
- 保持标题层级反映内容结构，不以字号决定标题级别。
- 为有信息意义的图片提供准确 `alt`，装饰图使用空 `alt=""`。
- 每个表单控件使用可见的 `label` 或等价的可访问名称。
- 交互控件使用原生 `button`、`a`、`input` 等元素。
- 保持键盘焦点可见，不用 `tabindex` 人为制造混乱顺序。
- 不只用颜色表达状态，错误信息应有文字说明。
- 音视频内容根据需要提供字幕、文字稿或替代说明。

ARIA 可以补充语义，但不能修复错误的交互模型。第一原则仍是优先使用原生元素。

## 字符、编码与实体

HTML 文件最终以字节传输，浏览器必须按字符编码把字节解码为 Unicode 文本。
现代页面应统一使用 UTF-8，并让 HTTP `Content-Type` 与 `<meta charset>` 保持一致。

HTML 实体或字符引用用于表示有语法意义或难以直接输入的字符：

```html
<p>比较：3 &lt; 5，商标：Q&amp;S。</p>
```

常见写法包括 `&lt;`、`&gt;`、`&amp;`、`&quot;` 和 `&#39;`。
字符引用不是通用安全方案；安全输出必须根据数据所处的 HTML、属性、URL、
JavaScript 或 CSS 上下文选择正确编码方式，详见安全篇。

## 验证与检查

开发时至少进行以下检查：

- 使用 HTML 验证器发现错误嵌套、重复 ID 和无效属性。
- 只用键盘完成链接、按钮和表单操作。
- 使用浏览器无障碍树或辅助功能检查器确认名称与角色。
- 在窄屏和宽屏上检查布局，不把移动端仅理解为缩小字体。
- 在网络面板确认状态码、响应头、请求方法和实际传输内容。
- 在元素面板确认浏览器最终生成的 DOM，而不是只查看模板源码。

验证器能发现结构问题，但不能判断文字是否准确、交互是否易懂或权限是否安全。

## 常见误区

- HTML 不是“低级编程语言”，它没有通常意义上的算法和控制流。
- `<br>` 表示换行，不是段落间距工具。
- `id` 在文档中应唯一，`class` 可以复用。
- 占位符 `placeholder` 不能替代持久可见的 `label`。
- 客户端表单验证改善体验，但不能替代服务端验证。
- 隐藏输入和禁用按钮都不是授权机制。
- HTTPS 保护传输，不会自动阻止 XSS、越权或恶意内容。
- 浏览器自动修复 HTML 不代表源码正确。

## 学习边界与 CTF

CTF Web 题常要求观察源码、DOM、请求参数、响应头、编码和浏览器行为。
这些技能也适用于正常开发与防御审计。本体系只在本地、赛事附件或明确授权环境中
讨论无害分析方法，不提供可直接用于未授权目标的攻击载荷或自动化利用。

## 学完后的目标

完成本系列后，应能做到：

- 准确说明 HTML 是标记语言，以及它与 CSS、JavaScript 的职责边界。
- 编写结构有效、语义清楚、具备基础可访问性的 HTML5 页面。
- 正确使用链接、媒体、表格和表单，并解释 GET 与 POST 的差异。
- 区分响应源码、解析后的 DOM、渲染结果与运行时修改。
- 使用元素、网络、控制台和查看源码功能定位常见问题。
- 理解 UTF-8、HTML 实体与不同输出上下文的转义要求。
- 说明客户端验证的边界，并采用服务端验证、输出编码和 CSP 等纵深防护。
