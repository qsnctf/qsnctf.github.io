# 什么是应急响应

应急响应（Incident Response，IR）是指组织在发现网络攻击、数据泄露、恶意程序感染、账号失陷或服务异常后，为了**确认事件、控制影响、消除威胁、恢复业务并避免再次发生**而采取的一系列有组织的行动。

简单来说，应急响应需要回答以下问题：

- 发生了什么？
- 事件从什么时候开始，攻击者如何进入系统？
- 哪些主机、账号、数据和业务受到影响？
- 如何尽快阻止影响继续扩大？
- 如何清除威胁并安全恢复业务？
- 如何保留证据、总结原因并防止事件再次发生？

## 应急响应的目标

应急响应的核心目标不是单纯“查杀病毒”或“重装系统”，而是在业务连续性、事件调查和安全风险之间取得平衡：

1. **控制损失**：尽快限制攻击范围，防止横向移动、数据外泄和持续破坏。
2. **查明原因**：还原攻击入口、攻击路径、受影响范围和事件时间线。
3. **清除威胁**：移除恶意程序、后门、异常账号、持久化机制和脆弱配置。
4. **恢复业务**：在确认环境可信后恢复系统、服务和数据。
5. **保全证据**：妥善保存日志、内存、磁盘和网络流量等证据，为后续调查与追责提供依据。
6. **持续改进**：通过复盘完善监控、流程、配置和人员协作机制。

## 常见安全事件

需要启动应急响应的情况通常包括：

- Web 网站被篡改、植入 WebShell 或遭遇恶意跳转。
- 服务器感染勒索软件、挖矿程序、木马或远程控制程序。
- 员工账号、邮箱、云平台密钥或管理员凭据泄露。
- 数据库中的敏感数据被未授权访问、下载或删除。
- 主机出现异常登录、异常进程、异常网络连接或权限提升行为。
- 业务遭受拒绝服务攻击，导致系统不可用。
- 内部人员违规操作或终端设备丢失造成数据泄露风险。
- 第三方组件、供应链或外部服务被入侵并影响本组织。

## 标准处理流程

应急响应通常分为六个阶段：

### 1. 准备

在事件发生前建立响应团队、联系人、处置预案、资产清单、日志策略、备份机制和取证工具。准备越充分，事件发生后的判断和处置越可靠。

### 2. 识别与分析

根据告警、日志、用户反馈和系统异常确认是否发生安全事件，并判断事件类型、严重程度、受影响资产和可能的攻击路径。此阶段需要记录时间线，避免仅凭单一现象下结论。

### 3. 遏制

阻止攻击继续扩散，例如隔离受感染主机、禁用失陷账号、封禁恶意地址、下线高风险服务或临时限制网络访问。遏制措施应兼顾业务影响和证据保护。

### 4. 根除

清除攻击者留下的恶意文件、进程、账号、计划任务、启动项和其他持久化机制，同时修复被利用的漏洞、弱口令和错误配置。只删除表面上的恶意文件，通常不能证明威胁已经消失。

### 5. 恢复

从可信备份恢复系统和数据，逐步恢复网络与业务访问，并持续监控是否仍有异常行为。恢复前应确认攻击入口已修复、凭据已轮换、系统完整性符合要求。

### 6. 复盘

整理事件报告，说明根因、时间线、影响范围、处置过程和改进措施。复盘结果应转化为可执行任务，例如补充检测规则、调整权限、完善日志和更新应急预案。

## 应急响应中的基本原则

- **先记录，后处置**：在条件允许时先记录系统时间、网络连接、登录会话、进程和告警信息。
- **先遏制，后恢复**：未控制攻击路径就直接恢复服务，可能导致系统再次被入侵。
- **保护现场与证据**：不要随意删除文件、清空日志或反复重启主机，以免破坏关键线索。
- **动态证据优先**：内存、进程、网络连接等易失信息可能在关机后消失，应优先采集。
- **最小化业务影响**：根据事件严重程度选择隔离、限流、切换或下线措施，避免处置本身造成更大损失。
- **全过程留痕**：记录操作人员、时间、命令、对象、结果和证据流转情况，保证过程可追溯。
- **统一沟通出口**：技术、管理、法务、公关和业务团队应按预案协作，避免未经确认的信息对外扩散。

## 应急响应与数字取证的区别

应急响应强调**快速控制风险并恢复业务**，数字取证强调**以规范方式获取、保存和分析证据**。两者经常同时进行，但目标侧重点不同。实际处置中，不能为了追求恢复速度而破坏关键证据，也不能因为过度取证而延误必要的遏制措施。

## 总结

应急响应是一项结合技术、流程和协作的系统性工作。一个完整的响应过程不仅要找到并清除当前威胁，还要明确攻击者如何进入、影响了哪些资产、是否仍有残留，以及组织应如何避免同类事件再次发生。其最终目标是将安全事件造成的损失降到最低，并提升组织面对下一次事件时的发现和处置能力。
